class hal_hw_interface.hal_io_comp.HalIO[source]¶

Bases: hal_hw_interface.ros_hal_component.RosHalComponent

HAL user component that sets up HAL pins connected to ROS

This component queries the ROS parameter server for hal_io/subscribe_pins, hal_io/publish_pins and hal_io/service_pins parameters, lists of pin names to pin data mappings.

The pin data must be a dictionary of pin configuration.

Example:

hal_io:
  subscribe_pins:
    enable:
      hal_type: BIT
      hal_dir: OUT
  service_pins:
    encoder_scale:
      hal_type: FLOAT
      hal_dir: OUT

The periodic update() function calls the pins’ update() functions, if any.

hal_hw_interface.hal_offset_mgr module¶

class hal_hw_interface.hal_offset_mgr.HalOffsetMgr[source]¶

Bases: hal_hw_interface.ros_hal_component.RosHalComponent

HAL user component managing zero offsets that persist across machine restarts

It reads joint names from the ROS parameter server, and creates two HAL pins for each: hal_offset_mgr.<joint>_offset and hal_offset_mgr.<joint>_fb-in, meant to be attached to corresponding pins of the joint’s offset component.

The machine start-up procedure should pull the hal_offset_mgr.load_params command pin high. The component will then copy saved offset values from the redis hal_offset_mgr/<joint>_offset keys to the corresponding hal_offset_mgr.<joint>_offset pins for each joint, and pull hal_offset_mgr.load_params low again.

The user interface should present the user with a ‘zero offsets’ switch that requests all offsets zeroed at the current position by pulling the hal_offset_mgr.zero_all_joints pin high. The hal_offset_mgr.enable pin must be low, or the component will refuse to do anything and issue an error message to that effect. Otherwise, the component will zero offsets by copying the hal_offset_mgr.<joint>_fb-in pins’ values to the hal_offset_mgr.<joint>_offset pins and the redis hal_offset_mgr/<joint>_offset keys for each joint.

Represents the offset of one joint.

By default, the pin name is <name>_offset, where <name> might simply be a joint name. This pin connects to an offset comp’s offset pin. Its value is persisted in a redis store key hal_offset_mgr/<name>_offset.

A second, complementary <name>_fb-in input pin connects to the offset comp fb-in pin. The zero_joint() function copies this value to the offset pin and updates the redis store.

hal_hw_interface.ros_hal_component module¶

synopsis:Boilerplate for HAL user components in ROS nodes
class hal_hw_interface.ros_hal_component.RosHalComponent[source]¶

Bases: hal_hw_interface.hal_obj_base.HalObjBase

Base class implementing a HAL user component in a ROS node

This class provides the skeleton of a ROS node that runs a Machinekit HAL user component. It does the tedious part of setting up the node and the component, running it, and shutting it down.

These components will often serve to connect the component’s HAL pins (which are connected in a larger HAL configuration from the hal_mgr ROS node) to ROS publishers, subscribers, or services. The hal_hw_interface.ros_hal_pin.RosHalPin classes provide an easy-to-use interface to set up these HAL pins connected to ROS.

Basic usage:

  • Create a subclass in a new module
  • Name the component by defining compname
  • Set up HAL pins and ROS communication in setup_component()
  • Handle periodic pin and ROS update logic in update()

A bare-bones example:

from hal_hw_interface.ros_hal_component import RosHalComponent

class MyRosHalComp(RosHalComponent):
    compname = 'my_comp'

    def setup_component(self):
        # Set up ROS HAL pins

    def update(self):
        # Do any updates, reading and setting pins

#!/usr/bin/env python

loadusr -Wn my_comp rosrun my_ros_hal_pkg my_ros_hal_comp
run()[source]¶

Run the ROS node/HAL component

Runs the component, calling the update() function in a loop until shutdown.

The rospy.Rate update rate will be taken from the ROS parameter <compname>/update_rate, defaulting to 10 Hz.

setup_component()[source]¶

Set up the ROS node and HAL component

This MUST be defined in subclasses to perform all set up tasks for the ROS node and HAL component. Much of this is automated in the hal_hw_interface.ros_hal_pin classes, which create HAL pins attached to ROS subscribers, publishers and services.

shutdown_component()[source]¶

Perform extra shutdown actions

update()[source]¶

The ROS node and HAL component update routine

This MUST be defined in subclasses. It is run periodically from the run() function, and should take care of all updates, such as reading or writing pins and communication with ROS.

class hal_hw_interface.ros_hal_pin.RosHalPinPublisher(name, hal_type=NOTHING, hal_dir=NOTHING, pub_topic=NOTHING, msg_type=NOTHING, last_value=None)[source]¶

Bases: hal_hw_interface.ros_hal_pin.RosHalPin

HAL pin with attached ROS publisher

This HAL pin is set with publishes its value on a ROS topic, <compname>/<name> by default. Its update() function should be called from the hal_hw_interface.ros_hal_component.RosHalComponent.update() function.

It will only publish its value when the value has changed. For FLOAT types, absolute and relative tolerances are read from the <compname>/absolute_tolerance and <compname>/relative_tolerance ROS parameters and default to 1e-9 each, respectively.

The publisher message type will be one of [Bool, UInt32, Int32, Float64] from the std_msgs ROS package, as appropriate for the HAL pin type.

Parameters:
  • name (str) – The HAL pin name
  • hal_type (hal_hw_interface.hal_pin_attrs.HalPinType) – HAL pin data type, one of ['BIT', 'U32', 'S32', 'FLOAT']
  • hal_dir (hal_hw_interface.hal_pin_attrs.HalPinDir) – HAL pin direction, one of ['IN', 'OUT', 'IO']
  • pub_topic (str) – ROS publisher topic
